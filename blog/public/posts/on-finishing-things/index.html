On Finishing Things

March 5, 2026 · 2 min read

I have, at last count, eleven unfinished side projects. A CLI tool for managing dotfiles that got to 80% before I decided to rewrite it in a different language. A home automation dashboard that works perfectly except for the bits that don’t. A blog engine — not this one, a different one, also unfinished.

I am, it turns out, very good at starting things.

The 80% problem

The first 80% of any project is the interesting part. You’re solving the core problem, making architectural decisions, writing the code that matters. You’re in flow. The last 20% is error handling, edge cases, documentation, packaging, the stuff that makes something actually usable by someone other than you at 2am on the specific machine you wrote it on.

The last 20% takes 80% of the time. This is not a coincidence.

Why we stop

Three reasons, roughly:

  1. The problem is solved — for you, personally, in your specific use case. The generalization work feels like overhead against a problem you’ve already stopped having.
  2. A shinier problem appeared. New language, new framework, new idea. The dopamine of starting is more available than the quieter satisfaction of finishing.
  3. Fear of judgment. A finished thing can be evaluated. An unfinished thing is still theoretically perfect.

The only fix I’ve found

Ut enim ad minima veniam, quis nostrum exercitationem ullam corporis suscipit laboriosam, nisi ut aliquid ex ea commodi consequatur.

Ship the 80% version. Declare it done. Write a short post about it. The act of publishing forces the kind of closure that private repositories never do. It doesn’t have to be perfect — it has to exist.

The progress bar on finishing side projects sits at 61% on my loading screen. That number is aspirational. But the fact that this post exists suggests the number is, very slowly, moving in the right direction.
